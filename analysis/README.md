# GPU Docking Pipeline
**QuickVina2‑GPU–Based High‑Throughput Docking (Single Receptor, Many Ligands)**

---

## 1. Purpose and Scope

This repository provides a two‑stage molecular docking pipeline designed for **high‑throughput docking of AI / machine‑learning–generated ligands** against a **single protein receptor** using **QuickVina2‑GPU**. Additionally, can be applied to any docking workflow with GPU access

### Primary Use Case
- **One protein receptor per docking run**
- **Many ligands** (from tens to millions) generated by ML/AI models and **predicted as potential binders**
- A **co‑crystallized ligand** is used to define the **binding pocket location** and **box dimensions** (for the docking search space)

The pipeline prioritizes **reproducibility, automation, GPU efficiency**, and **traceability of ligand identities** to help with fast analysis of results as well.

---

## 2. Pipeline Overview

```text
Raw inputs
├── receptor.pdb
└── ligands.sdf  (AI/ML-generated or other; may contain multiple molecules)

        │
        ▼
prep_data_files.py
 • Receptor preparation (PDB → PDBQT)
 • Ligand preparation (SDF → individual SDF + PDBQT)
 • Deterministic ligand IDs (title → InChIKey → SMILES hash) if none present
 • Parallel processing

        │
        ▼
Prepared docking directory
├── receptor_rec.pdbqt
├── pdbqt_ligands/
├── sdf_ligands/
└── ligand_index.tsv

        │
        ▼
qvina_gpu_dock.py
 • Define box from co‑crystal ligand (center + padded extents)
 • Write QuickVina2‑GPU config
 • Execute docking on shared GPU nodes

        │
        ▼
Docking outputs
├── poses (PDBQT)
├── logs
└── scores and metadata embedded in output PDBQT files
```

## 4. Script 1 — `prep_data_files.py`

### 4.1 Purpose
Prepares **all docking inputs** from raw structures within a single directory:
- Converts **receptor PDB files** into a **PDBQT receptor**
- Converts **sdf ligand files** in SDF into **individual SDF and PDBQT** files and folders
  - Created sdf files contained assigned ids for mapping later and reproducibility
- Enforces **stable ligand identity** assignment
- Supports **parallel ligand processing**

### 4.2 Inputs
Place the following in one directory (`INPUT_DIR`):
- One or more `*.pdb` receptor files
- One or more `*.sdf` ligand libraries (may contain multiple molecules each)


### 4.3 Outputs & Folder Layout
All outputs go to `--outdir` (or the input dir as default if not specified):

```text
outdir/
├── receptor_rec.pdbqt
├── sdf_ligands/
│   ├── ligand0.sdf
│   ├── ligand1.sdf
│   └── ...
├── pdbqt_ligands/
│   ├── ligand0.pdbqt
│   ├── ligand1.pdbqt
│   └── ...
└── ligand_index.tsv
```

### 4.4 Ligand Identity (Deterministic IDs for completeness and for analysis)
Ligands from ML pipelines may lack stable names. The script sets a **deterministic ID** for each ligand using this priority:
1. Existing SDF **title** (whitespace replaced with `_`)
2. RDKit **InChIKey**
3. **SHA‑1** hash of **canonical SMILES** (16 hex chars)

The chosen ID is written to:
- The **SDF title** of the split ligand and
- The **PDBQT** as a header line: `REMARK Name = <ligand_id>`

This enables robust **traceability**, **reproducible re‑runs**, and **post‑hoc mapping** back to the generator model.

### 4.5 Receptor Preparation Steps
For the single `*.pdb` file:
1. **Optional**: remove crystallographic waters (`HOH`).
2. Protonate with `pdb2pqr30 --ff=AMBER`.
3. Convert to receptor PDBQT via AutoDockTools (`prepare_receptor4.py`).

Output file name: `receptor_rec.pdbqt` (exact stem will follow the input file name).

### 4.6 Ligand Preparation Steps
For each molecule in each SDF:
1. Add hydrogens.
2. Generate 3D coordinates if missing.
3. Prepare AutoDock atom types/charges via Meeko.
4. Write `pdbqt_ligands/ligand<N>.pdbqt` and `sdf_ligands/ligand<N>.sdf`.
5. Inject `REMARK Name = <ligand_id>` into the PDBQT.

Processing of large libraries can be parallelized with `--nproc` workers.

### 4.7 Command‑Line Usage
```bash
python prep_data_files.py INPUT_DIR \
    --outdir OUTPUT_DIR \
    --del_water \
    --nproc 4
```

#### Arguments
- `INPUT_DIR` — directory containing `*.pdb` (one receptor) and `*.sdf` (ligands)
- `--outdir` — destination directory (default: same as `INPUT_DIR`)
- `--del_water` — remove water molecules from receptor
- `--nproc` — number of parallel workers for ligand prep

## 5. Script 2 — `qvina_gpu_dock.py`


### 5.1 Purpose
Runs **QuickVina2‑GPU‑2‑1** docking using:
- **One receptor** (`*_rec.pdbqt`)
- **Many ligands** (directory of ligand PDBQTs)
- **Reference (co‑crystallized) ligand SDF** to define the docking box

### 5.2 Reference Ligand (Co‑Crystal)
The **reference SDF** should correspond to a **co‑crystallized ligand** from the receptor’s experimental structure. The script uses its coordinates to set:
- **Center** of the search box (mean of atomic positions)
- **Box size** as the reference ligand’s extent plus a configurable **buffer**, with a per‑axis **minimum size** safeguard

These are both used to set a smaller search space then the entire protein surface, which would increase computational time, but still big enough to allow for the entire pocket to be analyzed.
Default parameters:
- `buffer = 7.0 Å`
- `min_size = 20.0 Å`

### 5.3 What the Script Generates
- A config file: `<config_prefix>_config.txt` containing receptor path, ligand directory, box center/size, OpenCL path, and GPU thread count
- Immediately runs: `QuickVina2-GPU-2-1 --config <config_file>`

> The QuickVina2‑GPU binary name is treated as **fixed** (`QuickVina2-GPU-2-1`) as that is the pattern for installation denoted by Qvina. If the installation path changes later, update `--qvina_bin` accordingly.

### 5.4 Command‑Line Usage
```bash
python qvina_gpu_dock.py \
    --receptor receptor_rec.pdbqt \
    --ligand_dir pdbqt_ligands \
    --reference_ligand sdf_ligands/ligand0.sdf \
    --qvina_bin /opt/qvina_gpu \
    --config_prefix run1
```

#### Arguments used to guide docking confguration
- `--receptor` — path to prepared receptor PDBQT
- `--ligand_dir` — directory containing ligand PDBQTs
- `--reference_ligand` — SDF used solely to define search box center and size
- `--qvina_bin` — directory containing the QuickVina2‑GPU binary (e.g., `/opt/qvina_gpu`)
- `--buffer` — Å padding added around reference ligand (default: `7.0`)
- `--min_size` — per‑axis minimum box size in Å (default: `20.0`) if ligand is small, box is still set to at least here
- `--threads` — GPU sampling threads for mp (default: `5000`) 
- `--config_prefix` — prefix for the generated config filename

### 5.5 End‑to‑End Example calling of preperation and docking
```bash
# Step 1: Prepare inputs
python prep_data_files.py raw_inputs \
    --outdir docking_inputs \
    --del_water \
    --nproc 4

or 

python prep_data_files.py raw_inputs \
    --outdir docking_inputs \
    --nproc 4

# Step 2: Run docking using the co‑crystal ligand to define the box
python qvina_gpu_dock.py \
    --receptor docking_inputs/protein_rec.pdbqt \
    --ligand_dir docking_inputs/pdbqt_ligands \
    --reference_ligand docking_inputs/sdf_ligands/ligand0.sdf \
    --qvina_bin /opt/VINA-GPU-2.1/QuickVina-GPU-2.1 \
    --config_prefix production \
    --threads 8000
```

## 6. Docking Outputs & Interpretation

The QuickVina2‑GPU run produces, typically in the ligand directory or working directory:

```text
pdbqt_ligands/
├── ligand0_out.pdbqt
├── ligand1_out.pdbqt
├── ...
└── log.txt (engine log; location may vary by build/config)
```

### 6.1 Scores & Poses
- Output PDBQT files can contain **multiple poses** per ligand, as defined in the configuration, usually ~10 is standard.
- Binding **affinity scores** (in kcal/mol) are embedded in the output PDBQT records.
- **Lower (more negative)** scores indicate stronger predicted binding affinity.

### 6.2 Mapping Back to Original Ligands
- Use `ligand_index.tsv` to map docking file stems (`ligand<N>`) to the stable `ligand_id` derived from title/InChIKey/SMILES.
- The `REMARK Name = <ligand_id>` line inside each output PDBQT provides an **inline identity check**.

> Tip: keep `ligand_index.tsv` and the final `config.txt` with your result set for full provenance and easier recollection of ran tests.

## 8. Troubleshooting
If you have erros during setup or usage, some of the common pitfals and potential fixes are listed here:

| Symptom                        | Likely Cause                                   | Mitigation                                                                             |
| ------------------------------ | ---------------------------------------------- | -------------------------------------------------------------------------------------- |
| No/poor poses for most ligands | Misplaced search box                           | Recheck reference ligand; enlarge `--buffer` or `--min_size`; visualize box.           |
| GPU not used / errors          | OpenCL/CUDA mismatch or missing device         | Verify GPU drivers; ensure QuickVina2‑GPU sees the device; confirm `--qvina_bin` path. |
| Some ligands skipped           | Existing output files or malformed SDF entries | Inspect console logs; validate SDF; re‑export problematic molecules.                   |
| Duplicated ligands             | Upstream generator emitted repeats             | Rely on `ligand_id` mapping; deduplicate by InChIKey before docking.                   |
| Receptor prep fails            | Missing `pdb2pqr30` or ADT scripts             | Ensure tools are installed and on PATH; try without `--del_water` to isolate issues.   |

## 10. Frequently Asked Questions
**Q: Can I use more than one receptor in a single run?**  
A: This pipeline is intentionally scoped to **one receptor per run**. Create separate run directories for additional receptors.

**Q: Can the box be defined manually?**  
A: Not in the current script; it auto‑derives from the co‑crystal. You can extend `qvina_gpu_dock.py` to accept manual `center_*` and `size_*` if needed.

**Q: Can I switch QuickVina versions?**  
A: The script currently assumes `QuickVina2-GPU-2-1`. You can change the binary name/path later, but it’s **fixed for now** by design.
